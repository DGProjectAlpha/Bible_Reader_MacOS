STEP 1: PER-PANE FLOATING NAVIGATION BAR
--------------------------------------------------------------

GOAL: Remove book/chapter navigation from the left sidebar and
move it into a floating overlay toolbar at the top of each
individual ReaderView pane. Each pane gets its own independent
navigation controls.

WHAT TO BUILD:
- New file: Views/PaneToolbar.swift
- A horizontal bar overlaid at the top of each ReaderView using
  ZStack + VStack alignment, floating OVER the verse scroll content
- The bar must use .background(.ultraThinMaterial) with
  RoundedRectangle(cornerRadius: 10) so it is semi-transparent
  and the verse text is visible scrolling behind it
- The bar must stay pinned to the top of the pane (not scroll away)

CONTROLS IN THE BAR (left to right):
1. Module picker — Menu { ForEach modules } label: { Text(abbreviation) }
   Switching module reloads the pane via bibleStore.navigate()
2. Book picker — Menu with two sections: "Old Testament" and "New Testament"
   Each section lists book names. Selecting a book navigates to chapter 1.
3. Chapter picker — Menu listing 1..N chapters for the current book
4. Verse picker — Menu listing 1..N verses for current chapter (optional,
   scrolls to verse via ScrollViewReader.scrollTo)
5. Spacer
6. Split Horizontal button — SF Symbol "rectangle.split.1x2"
   Calls bibleStore.addPane() with splitDirection: .horizontal
7. Split Vertical button — SF Symbol "rectangle.split.2x1"
   Calls bibleStore.addPane() with splitDirection: .vertical
8. Close pane button (X) — only visible when panes.count > 1

STYLING:
- All pickers use .menuStyle(.borderlessButton) for compact look
- Font: .callout weight .medium for labels
- SF Symbols: .font(.body) .foregroundStyle(.secondary)
- Buttons get .buttonStyle(.plain) with hover scale 1.05
- The entire bar: .padding(.horizontal, 12) .padding(.vertical, 6)

CHANGES TO EXISTING FILES:
- ReaderView.swift: wrap existing ScrollView in a ZStack. Place
  PaneToolbar at top with .frame(maxWidth: .infinity, alignment: .top)
  Add .padding(.top, 44) to the ScrollView content so verses
  don't hide behind the toolbar
- SidebarView.swift: REMOVE the module picker section, REMOVE the
  Old/New Testament book list and chapter grid sections entirely.
  Keep only: recent history section, and whatever remains after Step 2.
- ReaderArea.swift: no changes needed, pane management already works

APPLE API REFERENCES:
- Menu (macOS 11+): content + label pattern for dropdown
- .background(.ultraThinMaterial, in: RoundedRectangle(...)) for glass blur
- ScrollViewReader + scrollTo for verse jumping
- ZStack with alignment: .top for overlay pinning

STEP 3: DISABLE ANIMATIONS ON CHAPTER CHANGES AND SPLITS
--------------------------------------------------------------

GOAL: Remove all bounce/spring animations when changing chapters
or splitting/closing panes. Navigation should be instant — no
slide, no bounce, no transition. Panels opening/closing should
also be instant, not animated.

WHAT TO CHANGE:

1. ReaderView.swift — Chapter transitions:
   - REMOVE the .transition(.asymmetric(...)) on the verse content
   - REMOVE .animation(.spring(...), value: chapterId) or similar
   - Wrap chapter navigation state changes in withAnimation(nil) { }
     OR apply .animation(nil, value: pane.location) to the verse container
   - The chapter content should simply APPEAR — no slide, no opacity fade

2. ReaderArea.swift — Pane add/remove:
   - REMOVE .animation(.spring(duration: 0.35, bounce: 0.2)) on pane changes
   - Wrap bibleStore.addPane() and bibleStore.removePane() calls in
     withAnimation(nil) { } so new panes appear instantly
   - No transition on pane insertion/removal

3. ContentView.swift — Sidebar toggle:
   - REMOVE .animation(.spring(duration: 0.35, bounce: 0.2)) on sidebar toggle
   - Sidebar should appear/disappear instantly (no slide animation)
   - Use withAnimation(nil) { uiState.sidebarVisible.toggle() }

4. SidebarView.swift — DisclosureGroup animations:
   - Keep DisclosureGroup expand/collapse animations (these are fine,
     they're Apple-native and subtle)
   - But if any custom spring animations were added to sidebar content,
     remove them

5. General cleanup:
   - Search for all instances of .transition( and .animation(.spring
     in View files. Remove any that relate to navigation or panel toggling.
   - Hover effects (.scaleEffect on hover) may STAY — those are
     micro-interactions, not navigation animations

APPLE API REFERENCES:
- withAnimation(nil) { } — suppresses animation for state changes
- .animation(nil, value: x) — disables animation for specific value
- Transaction with disablesAnimations = true for broader suppression

STEP 4: FIX VERTICAL SPLIT VIEW RESIZING
--------------------------------------------------------------

GOAL: Both horizontal and vertical splits must be user-resizable
by dragging the divider. Currently only horizontal split can be
resized.

WHAT TO CHANGE:

1. ReaderArea.swift:
   - Verify that vertical pane stacks use VSplitView (not VStack).
     VSplitView (macOS 10.15+) provides built-in draggable dividers.
   - If the code uses VStack or manual GeometryReader for vertical
     splits, REPLACE with VSplitView { pane1; pane2 }
   - Each child in VSplitView needs .frame(minHeight: 150, idealHeight: 300)
     to allow resizing with a reasonable minimum
   - Each child in HSplitView needs .frame(minWidth: 200, idealWidth: 400)
   - Do NOT use .frame(maxHeight: .infinity) as it can lock the divider

2. Verify HSplitView children also have proper min/ideal/max constraints
   so horizontal dividers remain draggable

APPLE API REFERENCES:
- VSplitView (macOS 10.15+): arranges children vertically with
  user-draggable dividers. Children sized via .frame(minHeight:idealHeight:maxHeight:)
- HSplitView (macOS 10.15+): same pattern horizontally
- Both require children to have flexible frame constraints for
  the divider to be draggable

STEP 5: FIX FONT SIZE CONTROLS
--------------------------------------------------------------

GOAL: Font increase/decrease buttons must actually change the
bible verse text size in all reading panes.

WHAT TO INVESTIGATE AND FIX:

1. UIStateStore.swift — fontSize property:
   - Verify fontSize is @AppStorage-backed (persisted) and @Observable
   - It should be: var fontSize: Double = 16.0
   - If using @AppStorage directly, the @Observable wrapper may not
     publish changes. Fix: use a computed property that reads/writes
     @AppStorage, or use a didSet to sync

2. Toolbar/Settings font buttons:
   - Find where font +/- buttons are defined (likely toolbar or SettingsView)
   - Verify they modify uiState.fontSize directly:
     Button { uiState.fontSize += 1 } label: { Image(systemName: "textformat.size.larger") }
     Button { uiState.fontSize -= 1 } label: { Image(systemName: "textformat.size.smaller") }
   - Add bounds: max(10, min(40, fontSize))

3. VerseRow.swift — Apply font size:
   - The verse text must use .font(.system(size: uiState.fontSize))
   - Verify VerseRow reads @Environment(UIStateStore.self) var uiState
   - The verse number should scale proportionally: .font(.system(size: uiState.fontSize * 0.7))

4. If font size buttons are in the toolbar but the toolbar was removed
   during Step 2 cleanup, add them to PaneToolbar.swift (from Step 1)
   or to a global toolbar that persists:
   - Option A: Add A+/A- buttons to each PaneToolbar (per-pane font control)
   - Option B: Add to the main window toolbar (global font control)
   Recommend Option B — font size should be global, not per-pane.

5. SettingsView.swift — Font slider:
   - Verify the slider also writes to uiState.fontSize
   - The slider preview text should update live

APPLE API REFERENCES:
- @AppStorage for persisted user defaults
- .font(.system(size:)) for dynamic font sizing
- @Observable property changes trigger view updates automatically
